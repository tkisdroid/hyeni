    --   migration-time `created_at`, destroying chronological ordering for

    --   "actual creation time" (post-migration) and fall back to ORDER BY id.

    --   existing rows remain NULL (no backfill in this migration — live data
    --   invariant per CONTEXT.md D-06).
    -- • `user_id` references auth.users(id) with ON DELETE SET NULL to keep
    --   memos when a child account is deleted (audit log preservation).
    -- • `user_role` is plain text (no CHECK constraint) — Phase 5 SOS-01 may
    --   later tighten this; not in Phase 1 scope.
    -- • Idempotent via IF NOT EXISTS — safe to re-run on the Supabase branch.
    --
    -- Paired rollback: supabase/migrations/down/${TS}_reconcile_schema_drift.sql

    BEGIN;

    -- 1. memos.created_at — used by ORDER BY across RT-03 consumers.
    --    NULLABLE by design (D-06): NOT NULL DEFAULT now() would assign
    --    identical migration-time timestamp to every legacy row, destroying
    --    chronological ordering. NULL signals "unknown creation time" so
    --    consumers fall back to ORDER BY id for legacy rows.
    ALTER TABLE public.memos
      ADD COLUMN IF NOT EXISTS created_at timestamptz DEFAULT now();

    -- 2. memos.user_id — sender attribution (MEMO-03 placeholder; full wiring
    --    lands in Phase 4).
    ALTER TABLE public.memos
      ADD COLUMN IF NOT EXISTS user_id uuid
        REFERENCES auth.users(id) ON DELETE SET NULL;

    -- 3. memos.user_role — `'parent'` | `'child'` | NULL for legacy rows.
    ALTER TABLE public.memos
      ADD COLUMN IF NOT EXISTS user_role text;

    -- {OPTIONAL — only if Task 1 diff surfaced additional drift items that
    --  genuinely live in prod and are missing from tracked migrations.
    --  Example pattern:
    --
    --  ALTER TABLE public.<other_table>
    --    ADD COLUMN IF NOT EXISTS <col> <type>;
    --
    --  DO $$ BEGIN
    --    CREATE POLICY IF NOT EXISTS <policy_name> ON public.<other_table>
    --      FOR SELECT USING (...);
    --  EXCEPTION WHEN duplicate_object THEN NULL; END $$;
    --
    --  Each such addition MUST have a matching DROP in the down file.
    --  Only absorb drift — do NOT introduce new features or policy
    --  tightenings here. Those belong to later phases.}

    COMMIT;

    - Do NOT set a default value on `user_id` or `user_role` (no safe default exists; Phase 4 handles backfill via shadow-copy). Only `created_at` has `DEFAULT now()` so new inserts populate automatically while legacy rows stay NULL.
    - Do NOT run `UPDATE public.memos SET ...` in this migration — no backfill allowed in Phase 1 per CONTEXT.md boundary ("ONLY: file moves, new files, git tag, Supabase branch ops").
    - Do NOT DROP / RECREATE any existing table, policy, or index — additive only.
    - Do NOT include secrets, family IDs, or DB URLs in the file.

    If the Task 1 diff surfaced RLS policy drift that genuinely lives in prod but not in tracked migrations, absorb it here in the format shown in the `{OPTIONAL}` block above, using `DO $$ BEGIN ... EXCEPTION WHEN duplicate_object THEN NULL; END $$` to guard `CREATE POLICY` (Postgres does not support `CREATE POLICY IF NOT EXISTS` in all versions). Each absorbed item MUST have a matching DROP added to the down file in Task 3. If you add ANY additional statements beyond the three `memos` columns, append a comment `-- ABSORBED FROM DIFF: <description>` above each one so Task 3 and the reviewer can trace the pairing.
