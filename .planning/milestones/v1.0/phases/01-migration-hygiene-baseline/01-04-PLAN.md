<objective>
Capture three rollback anchors before any Phase 2+ work touches live data:

1. A full `pg_policies` snapshot of the production Supabase database (CSV + ordered SQL rendering) — the "what RLS looked like before any Phase 2 Stream C change" source of truth (per ARCHITECTURE.md §3.2: "BEGIN/COMMIT wrap ... no gap exists").
2. A non-secret environment metadata document recording the current VAPID public key, FCM project_id, and Supabase anon URL/key — the "what push infra is wired to TODAY" snapshot so Phase 2 Stream A can verify the pre-fix state and NOT rotate VAPID keys by accident (SUMMARY.md stack decision: "Do NOT rotate VAPID keys during redeploy").
3. A git annotated tag `push-notify-baseline-20260421` fixed to the current main HEAD — the "rollback this commit if Phase 2 Stream A breaks everything" anchor.

Purpose: Phase 2 Stream A (PUSH-01 ES256 gateway) ships in the next phase and will redeploy the `push-notify` Edge Function. If that redeploy causes a VAPID mismatch, the only safe recovery is bit-exact restoration — which requires knowing what was deployed BEFORE the change. This plan captures that baseline.

Output: Four files under `.planning/research/baselines/` + one git tag pushed to origin. Zero code changes. Zero DB changes.
