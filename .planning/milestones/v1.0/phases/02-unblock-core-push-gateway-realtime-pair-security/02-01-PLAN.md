<objective>
Close Phase 2 Stream A gateway: `push-notify` Edge Function returns HTTP 401 (`UNAUTHORIZED_UNSUPPORTED_TOKEN_ALGORITHM: ES256`) on every real user call because Supabase Auth rotated to ES256 asymmetric keys (2025-10-01 cutover) and v30 of the function doesn't verify ES256 JWTs in-function. Phase 1 edge-function log analysis (01-05-SUMMARY.md §5) confirmed v29 was POST 200 / v30 is POST 401 — this is a **code deployment regression**, not an infrastructure bug.

Rewrite the auth block in `supabase/functions/push-notify/index.ts` to use the official 2026 Supabase pattern `supabase.auth.getClaims(token)` — ES256 + kid rotation + JWKS caching transparent, ships in `@supabase/supabase-js@2.99.1` already (no dep bump). Redeploy as v31 via `npx supabase functions deploy push-notify --no-verify-jwt` (keeping the deploy flag per D-A02 to route around gateway bug supabase#42244).

Additionally, create the `push_idempotency` dedup table schema (D-A05) — Phase 2 only creates the table; Phase 3 P1-4 wires it into the dedup code path.

Purpose: Unblocks PUSH-02/03/04 (Phase 3), RL-02 (Phase 5 Stream B — remote_listen push trigger), KKUK-01..03 (Phase 5 Stream C — 꾹 push fallback), and SOS-01 (Phase 5 audit log). Without v31, every downstream push REQ is unverifiable. Also closes the body-trusted `senderUserId` exploit (PITFALLS P0-1 §"gateway-not-function 401" + PITFALLS §"security mistakes" row 1: push-notify accepts `senderUserId` from request body → spoofable).

Output:
- `supabase/functions/push-notify/index.ts` — two-client pattern (authClient for getClaims, db for service-role work); auth gate at top of Deno.serve handler; `senderUserId` derived from verified JWT claim, not body.

- Edge Function redeployed to production as v31 via Supabase CLI.
- Post-deploy smoke: curl with valid ES256 JWT → 2xx; curl with bogus JWT → 401.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → push-notify gateway | Untrusted `Authorization: Bearer <jwt>` header crosses Supabase edge gateway (deployed with `--no-verify-jwt` per D-A02 — gateway bug workaround) |
| gateway → push-notify function body | In-function `authClient.auth.getClaims(token)` verifies ES256 signature against JWKS; only verified `claims.sub` crosses into DB logic |
| push-notify → Supabase DB (service role) | After auth gate, service-role client performs DB writes; RLS bypassed (by design — function is the only authenticated API path for push) |
| client body → push-notify logic | `body.senderUserId` is NO LONGER trusted for `authenticated` callers; derived from `claims.sub` instead |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | push-notify `body.senderUserId` | mitigate | Derive `senderUserId` from verified JWT `claims.sub` (Task 2 Step 4); ignore body field for `authenticated` role; allow override ONLY for `service_role` caller (cron). |
| T-02-02 | Tampering | JWT Authorization header | mitigate | `auth.getClaims()` performs Web Crypto ES256 signature verification against Supabase JWKS; tampered tokens return `error` and 401 response (Task 2 Step 2). |
| T-02-03 | Repudiation | Push send actions | accept | Audit log for push actions is a Phase 5 SOS-01 scope item (sos_events table); Phase 2 does not add audit. Existing `pending_notifications` already records delivered/failed — deferred to Phase 3 PUSH-04 for telemetry enhancement. |
| T-02-04 | Information disclosure | FCM private key in logs | mitigate | PITFALLS §Pitfall 7 log scrubber NOT added in this plan (deferred — not in CONTEXT.md for Phase 2). Existing code already avoids logging `FCM_PRIVATE_KEY` directly (reads whole `FCM_SERVICE_ACCOUNT_JSON`). Accept residual risk for Phase 2; revisit in Phase 3 P1-4 if any log scrubbing requirement surfaces. |
| T-02-05 | Denial of service | Unauthenticated high-volume requests | mitigate | Gateway deployed with `--no-verify-jwt` still enforces Supabase platform-level rate limits; in-function getClaims returns 401 fast (single Web Crypto verify, <5ms) before any DB work. |
| T-02-06 | Elevation of privilege | Anonymous JWT impersonating paired child | accept | Anonymous child JWTs have valid `claims.sub`; derived `senderUserId` is the anon user's UUID. No privilege elevation possible (RLS at DB layer already scopes to `family_id`). |
| T-02-07 | Elevation of privilege | service_role JWT body override | accept | `callerRole === "service_role"` is only achievable if caller holds the service_role key (cron + other Edge Functions).
