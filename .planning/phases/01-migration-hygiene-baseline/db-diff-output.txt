## Finding 1 (primary — THIS IS WHAT PLAN 01-03 RECONCILES)

**`public.memos` in production is MISSING the three columns consumer code reads.**

### Prod schema (observed):

```
SELECT column_name, data_type, is_nullable, column_default
FROM information_schema.columns
WHERE table_schema = 'public' AND table_name = 'memos'
ORDER BY ordinal_position;

 column_name | data_type                   | is_nullable | column_default
-------------+-----------------------------+-------------+-------------------
 id          | uuid                        | NO          | gen_random_uuid()
 family_id   | uuid                        | NO          | NULL
 date_key    | text                        | NO          | NULL
 content     | text                        | YES         | ''::text
 updated_at  | timestamp with time zone    | YES         | now()
 read_by     | ARRAY                       | NO          | '{}'::uuid[]
```

### Missing columns (present in tracked-migration consumers / sister table `memo_replies`):

| Column       | Expected type | Expected nullable | Evidence                                                                 |
|--------------|---------------|-------------------|--------------------------------------------------------------------------|
| `created_at` | `timestamptz` | YES (NULL legacy) | 42703 in prod when consumer code does `ORDER BY created_at` on `memos`   |
| `user_id`    | `uuid` → `auth.users(id)` | YES (NULL legacy) | 42703 when consumer attributes memo to sender                    |
| `user_role`  | `text` (parent/child/NULL) | YES (NULL legacy) | 42703 when UI tries to render `memos.user_role` avatar distinction |

Sister table `public.memo_replies` DOES have all three and is the template the plan follows:

```
 column_name | data_type                   | is_nullable | column_default
-------------+-----------------------------+-------------+-------------------
 id          | uuid                        | NO          | gen_random_uuid()
 family_id   | uuid                        | NO          | NULL
 date_key    | text                        | NO          | NULL
 user_id     | uuid                        | NO          | NULL
 user_role   | text                        | NO          | 'child'::text
 content     | text                        | NO          | NULL
 created_at  | timestamp with time zone    | YES         | now()
```

Note: `memo_replies.user_id` is `NOT NULL` but `memos.user_id` MUST be NULLABLE for the
reconciliation — legacy `memos` rows pre-date any user-tracking and there is no safe
backfill value. Per CONTEXT.md D-06:
> "이미 데이터가 있으므로 NOT NULL 금지, 기본값 NULL"

This is the 42703 root cause documented in CONTEXT.md D-06 and PITFALLS.md line 150.

## Finding 2 (DEFERRED — NOT absorbed by Plan 01-03)

**Six RLS policies on `public.memos` show quals with inconsistent patterns** — evidence
of the two-SQL-dirs drift (loose `fix-*.sql` files applied over time without tracked
migrations). Plan 01-03's scope per the executor brief is the three missing columns
ONLY; non-memos DDL (and non-column drift on memos) is deferred unless the diff surfaces
it AND the executor surfaces it first. Captured here for downstream phase reference.

Observed policies on `public.memos` (six):

| Policyname      | cmd    | qual / with_check style                                      |
|-----------------|--------|-------------------------------------------------------------|
| `memo_sel`      | SELECT | `family_id IN (SELECT get_my_family_ids())`                 |
| `memo_ins`      | INSERT | `with_check: family_id IN (SELECT get_my_family_ids())`     |
| `memo_upd`      | UPDATE | `family_id IN (SELECT get_my_family_ids())`                 |
| `memo_del`      | DELETE | `family_id IN (SELECT get_my_family_ids())`                 |
| `memos_insert`  | INSERT | `with_check: family_id IN (SELECT fm.family_id FROM family_members fm WHERE fm.user_id = auth.uid())` |
| `memos_update`  | UPDATE | `family_id IN (SELECT fm.family_id FROM family_members fm WHERE fm.user_id = auth.uid())`             |

Two policy "families" coexist (`memo_*` using `get_my_family_ids()` helper vs `memos_*`
using inline `family_members` join). This duplication is real drift but:

1. Both families RESULT in the same access pattern (caller must share a family).
2. Tightening to a single family risks breaking one or the other access path.
3. Reconciling here without tracked tests in place would be a scope expansion beyond
   "migration hygiene baseline."

**Deferred to:** Phase 2 Stream C (`pair_code` / RLS hardening, per CONTEXT.md D-06
scope boundary "ONLY: file moves, new files, git tag, Supabase branch ops").

## Finding 3 (DEFERRED — NOT absorbed by Plan 01-03)

There is no tracked `CREATE TABLE public.memos` in `supabase/migrations/*.sql`. The
table is created by one of the archived loose SQL files (predates migration
discipline). Plan 01-03 does not attempt to retroactively author a CREATE TABLE for
`memos` — that is out of scope. The reconciliation migration uses `ADD COLUMN IF NOT
EXISTS` which is safe regardless of whether tracked migrations ever defined the table
(the columns go onto the live table).

---

## TO ABSORB (in Plan 01-03's up migration)

1. `ALTER TABLE public.memos ADD COLUMN IF NOT EXISTS created_at timestamptz DEFAULT now();` — NULLABLE.
2. `ALTER TABLE public.memos ADD COLUMN IF NOT EXISTS user_id uuid REFERENCES auth.users(id) ON DELETE SET NULL;` — NULLABLE, no default.
3. `ALTER TABLE public.memos ADD COLUMN IF NOT EXISTS user_role text;` — NULLABLE, no default.

Paired down file drops in reverse order (user_role → user_id → created_at).

## DEFERRED (NOT in Plan 01-03's scope)

1. `public.memos` RLS policy duplication (`memo_*` vs `memos_*` families) — Phase 2 Stream C.
2. Absence of tracked `CREATE TABLE public.memos` — backfill out of scope per CONTEXT.md D-03.
3. Broader schema diff (other tables) — Docker-less environment cannot run `supabase db diff`; a full diff sweep should happen in Plan 05 on the Supabase branch where Docker is available, or via manual inspection if further drift surfaces.
