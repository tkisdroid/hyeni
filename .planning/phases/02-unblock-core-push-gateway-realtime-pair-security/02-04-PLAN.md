<objective>
Close Phase 2 Stream C Wave 2 — RLS DELETE tightening on `family_members` (PAIR-03). Current production policy (captured in Phase 1 pg-policies-20260421.csv) allows children to self-DELETE their `family_members` row; this is the core PAIR-03 vulnerability — a lost-phone attacker keeps access until the child row is deleted, and children accidentally unpair themselves on logout. Tighten DELETE to parent-only.

Two concrete changes, both in a single migration (one BEGIN/COMMIT transaction per PITFALLS §Pitfall 3.2 to avoid the DROP-then-CREATE gap):
1. Drop the permissive `fm_del` policy
2. Create a restrictive `fm_del` policy that checks `EXISTS(parent row for auth.uid() in same family)` OR `families.parent_id = auth.uid()`

Rollback path: paired down file recreates the pre-Phase-2 permissive policy, captured verbatim from `pg_policies` pre-migration. Phase 1 baseline CSV is the cross-check.

Dependency on 02-03: the `join_family` RPC refresh (Plan 02-03) must land first so the RPC's behavior is consistent with the tightened DELETE. If 02-04 lands alone before 02-03, a child could still be removed by the old `join_family` path via re-pair (harmless — this tightening makes it strictly MORE secure, never less). However, we still serialize to preserve a clean, single-phase rollback surface.

Purpose: Satisfies PAIR-03 (child self-DELETE on `family_members` returns 403 / 0-rows; parent-initiated unpair still works via existing `unpairChild` helper). Unblocks Plan 02-05 UI (which exercises the parent path for PAIR-04 zombie cleanup — the UI calls `unpairChild`, which must continue to work post-tightening).

**Current (pre-Phase-2) fm_del policy** (per archive/_deprecated_fix-sync-final.sql:153-157; authoritative copy in Phase 1 baseline CSV):
```sql
CREATE POLICY "fm_del" ON family_members FOR DELETE
  USING (
    user_id = auth.uid()
    OR family_id IN (SELECT id FROM families WHERE parent_id = auth.uid())
  );
```
This allows **any authenticated user to delete their own row** (the `user_id = auth.uid()` branch) — the PAIR-03 vulnerability. Also allows the primary parent to delete any row in their family.

**New (Phase 2) fm_del policy** (D-C05 spec):
```sql
CREATE POLICY "fm_del" ON family_members FOR DELETE TO authenticated
  USING (
    -- Only parent role can delete. Covers:
    -- (a) Primary parent (families.parent_id = auth.uid())
    -- (b) Any parent co-member (family_members row with role='parent' for auth.uid())
    EXISTS (
      SELECT 1 FROM families f
       WHERE f.id = family_members.family_id
         AND f.parent_id = auth.uid()
    )
    OR EXISTS (
      SELECT 1 FROM family_members me
       WHERE me.family_id = family_members.family_id
         AND me.user_id = auth.uid()
         AND me.role = 'parent'
    )
  );
```

**Why `EXISTS` instead of IN-subquery:** Same effective semantics, but `EXISTS` is conventional for policy bodies and the Supabase docs recommend it for RLS subqueries (mildly more predictable plan stability at scale per PITFALLS §Performance Traps row 4).

    **Important — Postgres RLS simulation caveat:** RLS policies use `auth.uid()` which reads `request.jwt.claim.sub` OR parses `request.jwt.claims` JSONB.
