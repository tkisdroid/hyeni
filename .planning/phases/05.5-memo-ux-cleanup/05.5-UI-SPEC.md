Date label format:
- Same calendar day as today → "오늘"
- One calendar day before today → "어제"
- Otherwise → "4월 20일 월요일" (Korean `toLocaleDateString('ko-KR', {month:'long',day:'numeric',weekday:'long'})`)

#### 4b. Message Group Logic

Group consecutive messages from the same `user_id` where each message's
`created_at` is within 180 seconds (3 minutes) of the previous message from
the same sender. A new group starts when:
- Sender changes, OR
- Time gap > 180 seconds, OR
- A date separator falls between them.

Within a group:
- Avatar is shown only on the FIRST bubble of the group (top of group).
- Timestamp is shown only on the LAST bubble of the group (bottom of group).
- Middle bubbles: no avatar (replaced with 28px wide transparent spacer to
  maintain alignment), no timestamp.

#### 4e. Timestamp

```
<div>
  fontSize: 10px
  color: #9CA3AF
  marginTop: 4px
  textAlign: isMe ? "right" : "left"
```

Format rules:
- Age < 60 seconds: "방금"
- Age 1 min–59 min: "N분 전" (e.g. "5분 전")
- Age 1 h–23 h 59 min: "N시간 전" (e.g. "2시간 전")
- Age >= 24 h, same calendar year: "어제 오후 3:12" or "4월 15일 오후 3:12"
  using `toLocaleDateString('ko-KR')` + `toLocaleTimeString('ko-KR', {hour:'2-digit',minute:'2-digit'})`
- Age >= 24 h, different calendar year: "2025. 4. 15. 오후 3:12"

Timestamps are computed at render time from `r.created_at` (ISO string).
No real-time ticker needed — memo screen is not a live ticker UI.

#### 4f. Entry Animation

New bubbles appended to the list (optimistic inserts and real-time inserts)
receive a one-shot CSS animation via a style tag injected once on mount, or
via an inline animation string in the style prop.

```css
@keyframes bubbleIn {
  from { opacity: 0; transform: translateY(4px); }
  to   { opacity: 1; transform: translateY(0); }
}
```

Apply to newly added bubble wrapper:
```
animation: "bubbleIn 150ms ease-out forwards"
```

Detection of "new": use a `useRef` set of known IDs. Any ID not in the set on
the render cycle gets the animation class; add it to the set immediately after.

Reduced-motion: wrap animation assignment in:
```js
const prefersReducedMotion =
  window.matchMedia('(prefers-reduced-motion: reduce)').matches;
```
If true, skip animation entirely (no transform, no opacity transition).

The `<style>` tag injecting the keyframe must be rendered once inside
`MemoSection` (before the `role="log"` div) and must be idempotent (check
`document.getElementById('memo-bubble-anim')` before inserting).

## Interaction Contracts

### Scroll-to-Bottom

On new message append (optimistic insert or Realtime insert), scroll the window
to the bottom of `MemoSection` using:
```js
containerRef.current?.scrollIntoView({ behavior: 'smooth', block: 'end' });
```
where `containerRef` attaches to the outer MemoSection `<div>`.

Skip smooth scroll if `prefersReducedMotion` is true — use `behavior:'instant'`.

### Composer Focus State

Input border transitions from `#E5E7EB` to `#E879A0` on focus via `onFocus` /
`onBlur` handlers setting local `isFocused` state. Transition: none (border
color change only, no box-shadow to keep it lightweight).

### Send Button State

| Input state | Button appearance |
|-------------|------------------|
| Empty / whitespace only | background:#E5E7EB, cursor:default |
| Has text | gradient #E879A0→#BE185D, cursor:pointer |

`onClick` is always wired (button is never `disabled`). If input is empty,
`handleSend` returns early at the `!inputText.trim()` guard (existing logic).

### Read Receipt ("✓ 읽음")

Header-level only. Shown when `hasMessages && othersRead` (existing condition,
unchanged). No per-bubble read indicators. No per-bubble "읽음" text.

## Accessibility Contract

### Bubble Container

```
role="log"
aria-live="polite"
aria-label="메모 대화"
```

`role="log"` semantically marks the region as a live message log.
`aria-live="polite"` announces new messages to screen readers without
interrupting in-progress speech.

### Individual Bubble

Each bubble wrapper `<div>` gets:
```
role="article"
aria-label="{sender} {relative_time}에 보낸 메시지: {content}"
```

Sender label formula:
- `user_role === 'parent'` → "부모님"
- `user_role === 'child'` → "아이"
- `origin === 'legacy_memo'` → "예전 메모"
- me (isMe) → "나"

Example: `aria-label="부모님 5분 전에 보낸 메시지: 오늘 학원 잘 다녀와"`

### Composer

```
<input>
  aria-label={isParentMode ? "메모 입력" : "답글 입력"}
  aria-required="false"
  autoComplete="off"
  autoCorrect="on"
  spellCheck="true"

<button> (send)
  aria-label="메시지 보내기"
  type="button"
```

### Date Separator

```
<div role="separator" aria-label="{date_label}">
```

### Onboarding Toast

```
<div role="status" aria-live="polite" aria-label="메모 화면이 새로워졌어요">
```

Dismiss button:
```
aria-label="닫기"
```

### Tab Order

1. Composer `<input>` (primary focus on MemoSection open)
2. Send `<button>`
3. Toast dismiss button (if toast is visible)

Bubble content is not tab-focusable (decorative read-only log).

### Korean Screen Reader (VoiceOver / TalkBack)

- All `aria-label` strings are in Korean.
- Relative time strings ("5분 전", "방금") are grammatically correct Korean so
  TalkBack reads them naturally.
- Emoji in `aria-label` values are excluded; they appear only in visible content.

## Backward-Compatibility Note (onMemosChange Skip Logic Removal)

**UI-level impact of MEMO-FIX-02:**

The `onMemosChange` Realtime handler (App.jsx:4664-4678) currently has:
```js
if (newRow.date_key === dateKeyRef.current) return;
```

When the plan phase removes this guard (or narrows it to own-user rows only),
the `memos[dateKey]` state will update from incoming Realtime events. Because
the legacy textarea `<div>` block is being removed entirely by this phase,
there is no visible textarea element to show the stale state — the UI-level
ghost rendering is eliminated by removal, regardless of whether the Realtime
guard is narrowed first or simultaneously.

Recommended sequencing (planner note):
1. Remove the textarea DOM block first (MEMO-FIX-01 UI side).
2. Remove `upsertMemo` call sites (MEMO-FIX-01 logic side).
3. Remove / narrow `onMemosChange` skip guard (MEMO-FIX-02).

This order ensures there is no moment where a stale-updating textarea is
visible to the user, even if steps overlap in the same commit.

The `memos` state object and `onMemosChange` handler can remain in App.jsx
after this phase — they are not dead code yet because the `memos` poll
(`fetchMemos`) still runs on date navigation for archive display purposes.
Full removal deferred to v1.1 MEMO-CLEANUP-01 (drop `public.memos` table).
