Repo has:
- `public/manifest.json` (677 bytes, valid PWA manifest with icons)
- `public/icon-192.png` + `public/icon-512.png` (both ship)
- `public/sw.js` (PWA service worker)
- `index.html` line 6: `<link rel="manifest" href="/manifest.json" />`

Repo does NOT have:
- `vercel.json` (no file in repo root)

What Vite does out of the box:
- Any file under `./public/` is copied 1:1 into `./dist/` during `vite build` (no hashing, no transform).
- So `./dist/manifest.json` is produced by `npm run build` with byte-exact content of `public/manifest.json`.

What Vercel does WITHOUT a vercel.json:
- For a Vite SPA project it typically auto-detects and adds an implicit SPA rewrite (`/(.*) -> /index.html`). That rewrite will 200 on real files it finds first, but any auth edge rule, password protection, or deploy-protection setting will answer 403 BEFORE the static file is served.
- Alternative root cause: a previous Vercel password-protection or Branch Protection config may 403 `/manifest.json` while allowing the `index.html` HTML (because HTML is bypassed by deploy protection previews but JSON is not, in some edge configurations).

The fix is deliberately defensive: ship an explicit `vercel.json` that (a) sets content-type headers and (b) declares SPA rewrites that keep `/manifest.json` + `/sw.js` + icons out of the catch-all. If the root cause is deploy protection, the user will investigate separately (deferred in plan comments).

<task type="auto" tdd="false">
  <name>Task 2: Create vercel.json with explicit static + SPA routing</name>
  <files>vercel.json</files>
  <behavior>
    - `curl -I https://hyenicalendar.com/manifest.json` returns 200 after deploy.
    - `curl -I https://hyenicalendar.com/sw.js` returns 200 after deploy (defense in depth).
    - SPA deep links (e.g. `/whatever-route`) still resolve to `index.html`.
  </behavior>
  <action>
Create `vercel.json` in repo root with this content:

```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "headers": [
    {
      "source": "/manifest.json",
      "headers": [
        { "key": "content-type", "value": "application/manifest+json; charset=utf-8" },
        { "key": "cache-control", "value": "public, max-age=0, must-revalidate" }
      ]
    },
    {
      "source": "/sw.js",
      "headers": [
        { "key": "content-type", "value": "application/javascript; charset=utf-8" },
        { "key": "cache-control", "value": "no-cache, no-store, must-revalidate" },
        { "key": "service-worker-allowed", "value": "/" }
      ]
    },
    {
      "source": "/icon-(192|512).png",
      "headers": [
        { "key": "cache-control", "value": "public, max-age=604800, immutable" }
      ]
    }
  ],
  "rewrites": [
    { "source": "/((?!manifest\\.json$|sw\\.js$|icon-\\d+\\.png$|vite\\.svg$|assets/.*|api/.*).*)", "destination": "/index.html" }
  ]
}
```

Why each piece:
- `buildCommand` + `outputDirectory`: makes Vercel config explicit (avoids auto-detect drift).
- `headers[/manifest.json]`: guarantees correct MIME regardless of default detection. Chrome's installable-PWA check is strict about content-type.
- `headers[/sw.js]`: correct JS MIME + `service-worker-allowed: /` so the SW can control the whole origin. This is defensive — fixes the same class of bug for `/sw.js` if it shows up later.
- `rewrites`: negative-lookahead list that explicitly EXCLUDES static assets from the SPA catch-all. That catches the "rewrite ate my JSON" failure mode.
- Icon cache: 7 days immutable — safe since icons rarely change and filename is stable.

DO NOT add: auth bypass rules, `public: true`, Edge middleware. If the 403 is coming from Vercel deploy-protection (e.g. "Password Protection" toggle in Settings), config alone will NOT fix it — the user must disable that toggle, and Task 3 will surface it.
  </action>
  <verify>
    <automated>test -f vercel.json &amp;&amp; node -e "const c=require('./vercel.json'); if(!Array.isArray(c.headers)||!c.rewrites) process.exit(1); const hasManifest=c.headers.some(h=>h.source==='/manifest.json'); if(!hasManifest) process.exit(2);"</automated>
  </verify>
  <done>`vercel.json` exists, is valid JSON, has explicit `/manifest.json` header rule, and rewrites use a negative-lookahead that excludes `manifest.json`.</done>
