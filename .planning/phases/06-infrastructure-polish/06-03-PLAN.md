```sql
-- IDEMP-TTL-01 (Phase 6 Stream C) — pg_cron sweeper for push_idempotency.
-- Rows older than 24 hours are deleted hourly. Upstream table defined in
-- 20260421103838_push_idempotency_table.sql; this migration does NOT modify
-- that schema. Paired down: down/20260422000000_push_idempotency_ttl_cron.down.sql

BEGIN;

-- 1. Ensure pg_cron is available. Supabase allow-lists it on every project.
CREATE EXTENSION IF NOT EXISTS pg_cron;

-- 2. Sweeper function. SECURITY DEFINER so the cron runner (postgres role
--    in Supabase) can DELETE regardless of RLS state on push_idempotency.
--    Function is owned by postgres; pg_cron executes as postgres on Supabase.
CREATE OR REPLACE FUNCTION public.cleanup_push_idempotency()
RETURNS integer
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public, pg_temp
AS $$
DECLARE
  n_deleted integer;
BEGIN
  DELETE FROM public.push_idempotency
   WHERE created_at < NOW() - INTERVAL '24 hours';
  GET DIAGNOSTICS n_deleted = ROW_COUNT;
  RETURN n_deleted;
END;
$$;

COMMENT ON FUNCTION public.cleanup_push_idempotency() IS
  'IDEMP-TTL-01: deletes push_idempotency rows older than 24h. Called hourly by pg_cron job cleanup_push_idempotency.';

-- 3. Schedule. If a previous run left a job with the same name, unschedule it
--    first so re-running this migration is idempotent.
DO $cron$
DECLARE
  existing_jobid bigint;
BEGIN
  SELECT jobid INTO existing_jobid FROM cron.job
   WHERE jobname = 'cleanup_push_idempotency';
  IF existing_jobid IS NOT NULL THEN
    PERFORM cron.unschedule(existing_jobid);
  END IF;

  PERFORM cron.schedule(
    'cleanup_push_idempotency',
    '0 * * * *',
    $cmd$SELECT public.cleanup_push_idempotency();$cmd$
  );
END
$cron$;

COMMIT;
```

Notes the executor must respect:
- Schedule is `0 * * * *` (hourly). The planner considered `0 */6 * * *` (every 6h) but rejected it — 24h TTL + 6h granularity means rows can survive up to 30h, which violates the success criterion literally. Hourly keeps max-age at 25h, safely within tolerance. Cost: 24 row-scans per day on an indexed column; negligible.
- `SECURITY DEFINER` is deliberate: pg_cron on Supabase runs as `postgres`, but keeping it explicit protects against future Supabase changes that might run cron as a non-privileged role. `search_path` is locked to `public, pg_temp` (mandatory when using SECURITY DEFINER per Supabase best practice).
- DO NOT add RLS changes. `push_idempotency` RLS is out of scope.
- DO NOT `DROP EXTENSION pg_cron` in the down. Other cron jobs may rely on it.
- Filename format `YYYYMMDDhhmmss_…` matches the established convention (`20260421113053_…`).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| pg_cron runner -> push_idempotency | pg_cron runs as postgres; function is SECURITY DEFINER so RLS cannot block the sweeper. |
| Migration author -> live DB | MCP apply is a privileged path; codex review is the gate before apply. |
| Attacker with INSERT -> push_idempotency | Upstream push-notify is the only legitimate INSERTer (SEC-01 gate). Sweeper does not change this surface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-03-01 | Elevation of Privilege | SECURITY DEFINER function on public schema | mitigate | `SET search_path = public, pg_temp` locks resolution; function body is a single DELETE against a fully qualified table — no dynamic SQL, no user-provided args, no code-injection surface. |
| T-06-03-02 | Tampering | Another migration/DBA renames or re-schedules the cron job | mitigate | Forward migration is idempotent — re-running unschedules any prior `cleanup_push_idempotency` and re-registers. Drift is self-healing on next apply. |
| T-06-03-03 | Denial of Service | Sweeper deletes rows younger than 24h due to clock skew | accept | `created_at` uses server-side `NOW()` on insert and delete; both tick on the same clock. Skew is bounded by Supabase postgres clock monotonicity. |
| T-06-03-04 | Denial of Service | Sweeper row-lock storm during push spike | accept | `push_idempotency` has an index on `created_at`; 24h-old rows are rarely touched by live INSERTs. DELETE is short and non-transactional vs. push-notify INSERTs. Worst case: hourly 100-ms DELETE. |
| T-06-03-05 | Information Disclosure | Deleting rows that audit would have wanted | accept | `push_idempotency` is a dedup cache, not an audit log. Immutable audit for pushes lives elsewhere (`sos_events` SOS-01 etc.). Deletion has no compliance impact. |
| T-06-03-06 | Repudiation | Rollback forgets the cron job (leaving orphan) | mitigate | Down migration calls `cron.unschedule` first, then DROP FUNCTION. Both idempotent. |
| T-06-03-07 | Spoofing | Someone schedules a lookalike job name | accept | cron.job requires privileged role. Attack surface equivalent to any other postgres-privileged misuse, not new. |
</threat_model>
