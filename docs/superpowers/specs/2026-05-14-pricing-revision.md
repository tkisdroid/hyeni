## Why this revision

기존 spec (2026-04-18)에서 **"패밀리 단위 구독"**으로 명시된 design이 코드 구현(`src/lib/childSubscriptions.js`, `subscriptions.child_id` FK 등) 과정에서 **per-child 구독**으로 drift되어 있었다. 이 revision의 1차 목적은 spec drift 정정. 2차 목적은 1,500원/자녀 가격이 한국 시장 정상 밴드 대비 너무 싸고 구조도 통합 가치 명제와 충돌한다는 office-hours 진단을 반영한 가격 재설정.

---

## Problem Statement

혜니캘린더는 프로덕트가 100% 빌드 완료된 상태에서 정식 launch를 앞두고 있다. 코드에는 이미 페이월(자녀당 월 ₩1,500 / 연 ₩15,000, per-child)이 박혀 있다. 다음 두 가지가 검증되지 않은 채 가격이 결정되어 있는 상태:

1. **per-child 구조 drift** — 기존 spec은 패밀리 단위로 design 되었으나 구현에서 per-child가 됨. 다자녀 가족이 자녀 수만큼 누적 결제 (2명 = 3,000원, 3명 = 4,500원). 통합 가치 제안과 충돌.
2. **₩1,500 가격** — 한국 시장 정상 밴드(가족당 ₩4,900~7,900)의 1/3~1/5 수준. 침투 가격이 아니라 근거 없는 추측 가격일 가능성.

본 revision은 두 결정을 재정의하고, 솔로 개발자(아빠가 자기 아이를 위해 만든 앱) 정체성과 정합하는 무료/유료 구조 + 출시 이벤트 가격 + 종료 후 정상가를 확정한다.

## Demand Evidence

- **Q1 (강한 수요 증거):** "가족/지인이 필요한 앱이라 했다" — 카테고리상 **가장 약한 신호** (사회적 응답, 행동/지불 없음). 단독으로는 결정 근거 부족.
- **Q3 (구체 사용자):** 한 명을 picking 했을 때 "매일 사용하고, 본인이 말하지 않은 의존 행동이 있다" — 강한 신호로 격상. 그 행동:
  - 실시간 위치 엿보기 (불안 관리, 매일 다회 확인)
  - 일정/시간표/학원 입력 및 흐름 추적
  - 아이모드 / AI 동물 캐릭터 채팅을 통한 부모-아이 소통
  - **SOS/안전망은 매일 의존 행동에 포함되지 않음** — 의식적 안전망이지 매일 쓰는 product가 아니다.

## Status Quo

타겟 사용자가 혜니캘린더 도입 전 같은 문제를 풀던 방식:
- 키즈 스마트워치 + 통신비 (월 ₩6,000~12,000)
- Find My iPhone
- Google Family Link
- **여러 무료 도구를 듬성듬성 조합 + 통신비를 이미 지출 중**

→ **진짜 가치 명제는 "통합"**, **진짜 pain은 "앱 사이 왕복의 인지부담"**. 비용 추가가 아니라 비용 이관.

## Target User & Narrowest Wedge

- **사용자:** 한국 초·중등 자녀를 둔 부모. 이미 키즈워치/Family Link/카톡 가족방 등 분산 도구를 쓰고 있음.
- **Wedge:** "아이의 오늘 위치와 학원 루틴을 한눈에" — 가족 운영 OS로서의 매일 dashboard.

## Constraints

- **솔로 개발자 + 0 마케팅 예산** (출시 직전, 사용자 0명)
- **Google Play 수수료 30%** (실수령은 결제액의 70%)
- **인프라 (Supabase, GCP, 푸시 등) + 개발자 인건비** 지속가능성 필요
- **"아빠가 아이를 위해 만든 앱" 브랜드 정체성** — 인디/진정성 narrative를 가격과 정합되게 유지
- **주변소리듣기(remote audio listen)를 마케팅 표면에 두면 Google Play 스토커웨어 정책 리스크** + 부모/아이 신뢰 리스크. 표면에서 제외, 결제 후 깊은 설정 항목으로 위치.

## Premises

- **Premise A — 무료 티어를 강하게 유지한다.** Funnel 상단 acquisition 확보가 침투 전략의 핵심. (단, 출시 후 90일 conversion이 0.5% 미만이면 premise A 재검토 trigger.)
- **Premise B — 자녀별이 아니라 가족 단위(per-family) fixed 구독.** 기존 spec의 원래 의도이자 통합 가치와 정합. 코드 drift 정정.
- **Premise C — ₩1,500/월은 시장 신호상 너무 싸다는 진단에 동의하되, "아빠가 만든 앱"이라는 인디 브랜드 narrative와 침투 전략을 살리는 공격적 가격을 선택.** 시장 정상가(₩4,900~7,900)에 정면 진입하지 않고, 절반 이하의 acquisition 우선 가격으로 시장 진입.

## 추가 design 원칙 (Codex cold read 통찰)

- **전환 드라이버는 1개 문장으로 좁힌다.** "아이의 오늘 위치와 학원 루틴을 한눈에." 8개 페이월 기능을 같은 무게로 노출하는 "기능 백화점 함정"을 피한다.
- **주변소리듣기는 결제 후 설정 깊은 항목으로만 배치.** 마케팅/페이월 노출 텍스트에서 제외 (스토커웨어 정책 + 신뢰 리스크).
- **포지셔닝은 "안전 앱"이 아니라 "가족 운영 OS / 매일 보는 가족 대시보드".**

## Approaches Considered

### Approach A — Single Tier ₩4,900 (시장 정상가 진입)
- 가족 월 ₩4,900 / 연 ₩49,000, 7일 무료체험.
- 한국 standalone 앱 시장 정상가에 정면 진입.
- 거부 사유: 솔로 개발자 인디 정체성 미반영, 초기 acquisition velocity 부족 우려.

### Approach B — 2단계 Basic/Plus (Codex 권고)
- Basic ₩5,900, Plus ₩7,900.
- ARPU 최대화 우선.
- 거부 사유: 0 사용자 단계에서 2단계 구조는 어느 단이 잘 팔리는지 검증 데이터 부족. 결정 부담으로 conversion drop 위험.

### Approach C — 초기 이벤트 ₩1,900 (선택)
- 가족 월 ₩1,900 / 연 ₩19,000, 이벤트 기간 lifetime locked-in.
- "아빠가 만든 앱"이니까 정상가의 절반 이하 + 자녀 수 무관 가족 단위.
- 침투 가격 + 인디 정체성 정합.
- 종료 조건: 결제 가족 1,000 도달 또는 출시 후 12개월 중 빠른 시점.

## Recommended Approach: Approach C — 초기 이벤트 ₩1,900

### 무료 티어 (premise A 유지, 현재 코드 그대로)
| 기능 | 무료 |
|---|---|
| SOS 송수신 | ✅ |
| 1회성 위치 (location_one_shot) | ✅ |
| 실시간 음성 메시지 | ✅ |
| 오늘 일정 보기 | ✅ |
| 실시간 위치 (5분 지연) | ✅ (UI에 "5분 지연" 명시) |
| 자녀 연동 | 1명까지 |
| 위험지역 (geofence) | 1개까지 |
| 위치 이력 보관 | 30일 |

### 유료 티어 — "초기 이벤트" 기간 ₩1,900 / ₩19,000
| 기능 | 유료 |
|---|---|
| 실시간 위치 (지연 없음) | ✅ |
| 다자녀 연동 | 무제한 |
| 학원 일정 + 동선 | ✅ |
| 자주 가는 장소 (saved places) | 무제한 |
| 위험지역 (geofence) | 무제한 |
| AI 빠른 일정 입력 (음성, 메모 분석) | ✅ |
| 장기 위치 이력 | 30일 이후도 보관 |
| 주변소리듣기 (remote audio) | ✅ (설정 깊은 곳에만 노출, 마케팅 표면 미노출) |

### 결제 단위
- **가족 1개당 1구독** (자녀 수 무관) — 기존 spec의 원래 의도, 코드 drift 정정.
- Google Play 구독: 월 ₩1,900 / 연 ₩19,000

### 체험
- **7일 무료체험** (카드 등록형) — 기존 코드와 동일 유지.

### 전환 메시지 (단일 문장 원칙)
**"아이의 오늘 위치와 학원 루틴을 한눈에."**

페이월 화면, 스토어 설명, 마케팅 카피 모두 이 한 문장을 anchor로. 8개 기능을 분산 노출하는 "기능 백화점" 금지.

### "초기 이벤트" 종료 조건 (default 권장값, 사용자 수정 가능)
- **결제 가족 1,000 도달** 또는 **출시 후 12개월** 중 빠른 시점에 종료.
- 종료 이후 신규 결제자는 **정상가 월 ₩4,900 / 연 ₩49,000**으로 전환 (시장 정상 밴드 진입).
- 이벤트 기간 결제 가족: **₩1,900/₩19,000 lifetime locked-in** (Early Adopter 배지의 진짜 의미 = 영구 우대가).

### 추가 측정 trigger (premise A 검증)
- 출시 후 30/60/90일 누적 무료→유료 conversion 측정.
- **90일 누적 conversion < 0.5%**이면 premise A (강한 무료) 재검토. 무료 티어 좁히기 (예: 실시간 위치 5분 지연 → 위치 1일 1회만, 또는 SOS 시에만) alternative 재검토.

## Open Questions

1. **iOS 출시 시점에서의 App Store 수수료/가격 정합.** 현재 Google Play 기준으로만 가격이 설정됨. iOS 출시 시 30% 수수료 + 환율 변동 대응 가격 정책 필요.
2. **"초기 이벤트" 종료 시 신규 가입자에게 정상가(₩4,900/₩49,000) 적용 시 — 가격 인상 announcement 정책.** 종료 D-30, D-7, D-1 알림 + Press release 필요 여부.
3. **연 결제자 환불 정책.** Google Play 연 결제는 3일 이내 환불 정책이지만, 이벤트 lifetime grandfathered 약속의 법적/계약적 보호 형태 (이용약관 명시).
4. **아이모드 / AI 동물 캐릭터 채팅의 페이월 위치 미정.** 매일 사용 행동에 포함되는데 현재 `paywallCopy.js`에 명시되지 않음. AI_ANALYSIS(빠른 일정 정리)에 통합되는지, 별도 SKU인지 확인 필요.
5. **per-child → per-family 마이그레이션.** 기존 spec(2026-04-18)이 family-unit으로 design 되었음에도 코드는 `subscriptions.child_id` FK로 구현됨. `family_subscription` 테이블 + `subscription_tier` 트리거가 기존 spec에 정의되어 있었는지, 또는 신규 도입인지 확인 필요. 마이그레이션 SQL + 기존 결제자 데이터 정합 필요.

## Success Criteria

### Launch + 90일 (premise A 검증 게이트)
- 무료 사용자 누적 ≥ 5,000 가족
- 누적 conversion rate ≥ 0.5%
- 결제 가족 ≥ 25명 (초기 이벤트 기간 lifetime locked-in 확보)

### Launch + 6개월
- 무료 사용자 누적 ≥ 20,000 가족
- 결제 가족 ≥ 200명
- "초기 이벤트" 종료 trigger 준비 (1,000 도달 예측 / 12개월 진행 결정)

### Launch + 12개월 (이벤트 종료 시점)
- 결제 가족 ≥ 1,000 (초기 이벤트 quota 충족)
- 월 매출 ≥ ₩1,330,000 (1,000 가족 × ₩1,900 × 0.7 수수료 후) — 솔로 개발자 부분 수익화
- 정상가 ₩4,900 전환 후 신규 결제 churn ≤ 30%

### 정성 지표
- 페이월 노출 → 결제까지의 funnel에서 "아이의 오늘 위치와 학원 루틴을 한눈에" 메시지의 click-through 측정.
- 다자녀 가족 (자녀 2+) 비중 — 가족 단위 가격의 진짜 가치 검증.

## The Assignment

**출시 전 한 가지 실제 행동:**

페이월 화면을 "아이의 오늘 위치와 학원 루틴을 한눈에" 한 문장으로 다시 그리고, 그 페이월을 친척/지인이 아닌 **모르는 부모 3명**에게 직접 보여주세요. 카톡으로 페이월 스크린샷 전송이 아니라, **실제 앱을 그들 핸드폰에 베타로 설치**하고, **7일 무료체험 가입 화면까지 진행하는지** 옆에서 (말없이) 관찰하세요.

- 가입했다 → 그 사람이 1주 후 결제 전환했는지 확인 (real demand signal)
- 가입 안 했다 → 어디서 멈췄는지 (페이월 카피? 가격? 카드 등록? 신뢰?)
- 다운로드도 안 했다 → ₩1,900가 문제가 아니라 acquisition 자체가 문제

이 한 행동이 출시 후 90일 conversion 목표 0.5%를 prediction할 수 있는 유일한 실측 데이터입니다. 지금 가족/지인의 "필요한 앱이다"보다 1,000배 더 가치 있는 정보가 출시 전에 들어옵니다.

## What I noticed about how you think

- 옵션 A/B/C/하이브리드 4개를 제시했을 때 **하나도 안 고르고 새 방향("아빠가 아이를 위해 만든 앱 컨셉 + 공격적 가격")을 직접 제시**했습니다. 평범한 founder가 안 하는 행동입니다. 자기 브랜드 정체성을 명확히 인지하고 있고, 외부 권고(나의 추천 A, Codex의 Ideal B)에 default로 끌려가지 않는다는 신호.
- "freemium funnel 상단 sealed deal 차는 필수, 무료를 줄이면 시장도 안 열린다"고 premise A를 명시적 reasoning으로 방어한 부분 — 단순히 "내가 원해서"가 아니라 **모델을 가지고 push back**했습니다. office-hours 진단에서 가장 드물게 보는 founder signal.
- 그러나 demand evidence 단계에서 첫 답이 "가족/지인이 필요한 앱이라 했다"였다는 점. 가장 약한 신호를 가장 먼저 입에 올렸다는 것은, 평소에 그 답으로 자신을 설득하고 있을 가능성을 시사합니다. The Assignment가 그 자리를 채우기 위한 실측 행동입니다.
- 가격을 ₩1,500에서 거의 변화 없이 ₩1,900로 잡은 결정 — 이건 "이미 코드에 박힌 ₩1,500이라는 sunk anchor에서 멀리 떠나지 않으려는 무의식적 보수성"일 수도, "공격적 침투의 의도적 선택"일 수도 있습니다. 90일 데이터가 어느 쪽인지 답해줄 겁니다.
- 기존 spec이 family-unit으로 design 되어 있었는데 구현이 per-child로 drift된 사실을 office-hours 세션이 끝날 무렵에야 발견했습니다. 이건 founder가 자기 spec과 코드 사이의 미세한 일관성 손실에 대해 brain bandwidth를 다 못 쓰고 있다는 signal이고, 솔로 개발자 단계에선 자연스럽습니다. 다만 launch 전에 한 번 spec vs 코드 audit을 하는 게 안전합니다.
